### Key Il2Cpp Differences (from MelonWiki il2cppdifferences.md)

- **No transpiler patches** - Il2Cpp runs native code, not IL. Only Prefix and Postfix Harmony patches work.
- **Type casting** - Use `.TryCast<T>()` or `.Cast<T>()` from `Il2CppInterop.Runtime`, not standard C# casts.
- **String handling** - Methods that expect `System.String` may need `Il2CppSystem.String`. Explicit casting handles conversion.
- **Custom MonoBehaviours** - To inject a class into Il2Cpp you must: inherit a non-abstract Il2Cpp class, implement an `IntPtr` constructor, apply `[RegisterTypeInIl2Cpp]`, and provide derived constructors.
- **Delegates/Events** - Use `Il2CppSystem.Action` instead of `System.Action`. Events lose `+=`/`-=`; use generated `add_`/`remove_` methods instead.
- **Coroutines** - Use `MelonCoroutines.Start()` instead of `MonoBehaviour.StartCoroutine()` since MelonMod is not a MonoBehaviour.
- **Harmony patches must be static** - All prefix and postfix patch methods must be static methods.

### Harmony Patching (from MelonWiki patching.md)

Two approaches:

1. **Attribute-based**: Apply `[HarmonyPatch]` attributes to static patch classes. MelonLoader auto-discovers and applies them - no manual `PatchAll()` needed.
2. **Manual**: Call `Patch()` on your mod's `HarmonyInstance` property with explicit `MethodInfo` and `HarmonyMethod` objects.

Supported patch types: Prefix, Postfix, Finalizer. Transpilers do NOT work in Il2Cpp.

YgoMaster is a full offline server emulator for Master Duel. Its LiveMods feature allows a small set of client-side mods to be injected into the live game via process injection (not MelonLoader). The LiveMods documentation is entirely user-facing and does not document the UI system, ViewControllers, or ElementObjectManager at all. ### Rush Frames Mod (MelonLoader + Harmony reference)

- **GitHub**: https://github.com/Nauder/ygomd-rush-frames

This mod is the closest publicly available reference for the same technical stack as BlindMode: MelonLoader + Harmony on Master Duel targeting .NET 6. It patches visual card frame elements using Harmony postfixes. No source exploration was done but the structure (MelonLoader mod, Harmony patches) is confirmed.

## 3. API Reference: Does One Exist?

No official Konami/YgomGame API reference exists. The game is closed-source and Il2Cpp-compiled.

The closest thing to an API reference is `dump.cs.txt` (see section 4), which provides a complete type dump generated by Il2CppDumper. This IS the de facto API reference for this mod.

There is no community-maintained wiki or documentation for the `YgomSystem`/`YgomGame` type hierarchy beyond what individual modders have reverse-engineered themselves.

## 4. dump.cs.txt - Local File Analysis

**File exists at**: `dump.cs.txt` in the repo root (gitignored).

**Size**: 917,364 lines - this is a very large file covering all assemblies loaded by the game.

**Format**: Il2CppDumper output. The header lists all 73 game assemblies with their starting TypeDefIndex values. The body contains pseudo-C# class/method stubs for every type in the game.

**Primary game assembly**: `Assembly-CSharp.dll` (Image 0, starts at TypeDefIndex 0) - this contains all Konami's game code.

### Key game assemblies also present

- `Assembly-CSharp-firstpass.dll` (Image 31) - third-party/plugin code bundled with the game
- `YgomDebug.Network.dll` (Image 70) - debug networking
- `Unity.TextMeshPro.dll` (Image 9), `UnityEngine.UI.dll` (Image 12), `spine-unity.dll` (Image 15)

### Namespaces found in Assembly-CSharp

### Key types confirmed in dump (with TypeDefIndex)

**`ElementObjectManager`** (TypeDefIndex 765, `YgomSystem.ElementSystem`):
- Fields: `serializedElements` (ElementObject[] - the array used for label lookups), `elements` (SortedDictionary), `labelCounter`
- Methods: `GetElement(string label)` returns GameObject, `GetElement<T>(string label)` generic version, `GetLabels()`, `IsExistsLabel()`, `SetElement()`, `SetCallback()`, `GetNestedElement()`
- Note: `GetLabels()` returns `SortedDictionary.KeyCollection` which fails in Il2Cpp interop - the mod correctly iterates `serializedElements` array instead.

**`ViewController`** (TypeDefIndex 1392, `YgomSystem.UI`):
- Key methods: `OnBack()` (virtual), `OnFocusChanged(bool)` (virtual), `OnResult()` (virtual), `PushViewController()`, `PopViewController()`, `SendResult()`, `SendBack()`
- Fields: `manager` (ViewControllerManager), `acceptBack`, `Args` (Dictionary)

**`ViewControllerManager`** (TypeDefIndex 1401, `YgomSystem.UI`):
- Extends `ViewController`

**`BaseMenuViewController`** (TypeDefIndex 4682, `YgomGame`):
- Extends `TweenViewController`
- Key field: `m_View` (ElementObjectManager) - the primary UI element container
- Key virtual method: `OnCreatedView()` - lifecycle hook called after view assets are loaded. This is the primary patch point.
- Also has: `m_ViewCreater`, `m_AdditionalTweenTarget`, `m_TextGroupLoadHolder`

**`DialogViewControllerManager`** (TypeDefIndex 4700):
- Extends `ViewControllerManager`

**`ActionSheetViewController`** (TypeDefIndex 4454):
- Extends `SelectDialogViewControllerBase<string, ActionSheetViewController.EntryData[], int>`
- Implements `IBokeSupported`
- Has inner types: `EntryData`, `EntryButtonsScrollWidget`, `EntryButtonWidget`, `EntryButtonWidgetToggle`

**`SelectionItem`** (TypeDefIndex 1176, `YgomSystem.UI`):
- The focusable button/item base class
- Key property: `isSelected` (bool, backed by `<isSelected>k__BackingField`)
- Callbacks: `onSelectedCallback`, `onDeselectedCallback`, `onPointerClickCallback`, etc.
- Navigation links: `upItem`, `downItem`, `leftItem`, `rightItem` (all SelectionItem refs)
- Fields: `_selectable`, `_interactable`, `defaultItem`, `_priority`

### Total type count

The dump contains approximately 8,189 public types across all assemblies.

## 5. Key Findings for This Mod

### What works well

- The dump.cs.txt is a comprehensive type reference. It can be grepped directly for any class, method, or field name to understand the game's API surface.
- The `serializedElements` array approach for EOM lookup is validated: the dump shows `GetLabels()` returns `SortedDictionary.KeyCollection` which is the problematic Il2Cpp type, confirming why the array iteration workaround is necessary.
- `OnCreatedView()` as a patch point is well-chosen - it is a virtual method on `BaseMenuViewController` that fires reliably after `m_View` is populated.
- Harmony Prefix/Postfix patches are the correct approach (transpilers confirmed unsupported in Il2Cpp).

### Gaps / No documentation exists for

- Which `serializedElements` label strings exist on each specific ViewController - these must be discovered by runtime logging or grepping dump.cs.txt for `[SerializeField]` string fields.
- The exact timing/order of the ViewController lifecycle (when `m_View` is non-null, when `serializedElements` is populated, async loading patterns) - documented only through trial and error in the mod's own MEMORY.md.
- No community wiki or forum thread documents the `YgomSystem.UI` or `YgomSystem.ElementSystem` APIs.

### Recommended references for future development

1. `dump.cs.txt` (local) - grep for any type or method before coding
2. [MelonLoader Il2Cpp differences](https://github.com/LavaGang/MelonWiki/blob/master/docs/modders/il2cppdifferences.md) - for interop issues
3. [MelonLoader Harmony patching](https://github.com/LavaGang/MelonWiki/blob/master/docs/modders/patching.md) - for patch syntax
4. [ygomd-rush-frames](https://github.com/Nauder/ygomd-rush-frames) - only known public MelonLoader+Harmony Master Duel mod reference
5. [GameBanana Master Duel](https://gamebanana.com/games/17161) - community mod distribution hub, may have forum threads with implementation hints
